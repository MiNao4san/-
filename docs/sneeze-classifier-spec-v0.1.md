# くしゃみ判別アプリ 仕様書（初版）

## 1. 文書情報
- **文書名**: くしゃみ判別アプリ仕様書
- **版数**: v0.1
- **対象**: 初版
- **公開形態**: GitHub Pages
- **実行方式**: ブラウザアプリ

## 2. アプリ概要
ユーザーがマイクで録音したくしゃみ音をもとに、
**「おじさん寄り」または「上品寄り」** のいずれかに判定する Web アプリを提供する。

本アプリは初版では **ブラウザ内で録音・判定・保存を完結** させる構成を基本とする。
データ保存が必要な場合は `LocalStorage` を使用する。

## 3. 目的
- くしゃみ音を簡易に録音し、分類結果を表示する
- 過去の判定履歴を端末内で確認できるようにする
- GitHub Pages 上で静的ホスティング可能な構成にする
- サーバー常駐を前提としない

## 4. 判定対象の定義

### 4.1 分類クラス
判定クラスは以下の2値とする。
- おじさん寄り
- 上品寄り

### 4.2 判定方式
本アプリは **音声認識（文字起こし）ではなく、音声分類アプリ** として設計する。
したがって、入力音声の音響特徴に基づいてクラス分類を行う。

### 4.3 ラベル定義
学習用ラベルの基準は、以下とする。
- 複数人アンケートによる多数決ラベル

ただし、初版アプリ内では多数決ラベル収集を行わない。
学習データの作成・アンケート収集・ラベル確定は **別運用** とする。

## 5. スコープ

### 5.1 初版の対象機能
- マイク入力による録音
- 録音音声に対する分類
- 判定結果の表示
- 判定履歴の保存・表示
- 録音音声保存のオン/オフ選択
- GitHub Pages で公開可能な静的構成

### 5.2 初版の対象外
- 音声ファイルアップロード
- 複数人アンケート収集機能
- サーバー側での集計・モデル再学習
- アカウント機能
- クラウド同期
- 結果共有機能
- 文字起こし機能

## 6. 想定利用環境
- **対応端末**: PCブラウザ / スマホブラウザ 両対応
- **利用形態**: 単一ユーザーが端末上で利用
- **公開先**: GitHub Pages
- **通信要件**:
  - アプリ本体の配信時はネット接続を前提
  - 判定処理自体はブラウザ内完結を基本とする

## 7. システム構成

### 7.1 構成方針
- フロントエンドのみの静的アプリ
- サーバーサイド処理なし
- データ保存は `LocalStorage`
- 推論はクライアントサイドで実行

### 7.2 配布構成
GitHub Pages 上に以下を配置する想定。
- HTML
- CSS
- JavaScript / TypeScript ビルド成果物
- 必要に応じてモデルファイル
- 必要に応じてライブラリ同梱ファイル

### 7.3 外部依存
依存ライブラリ／モデルの読み込み方法は以下を許容する。
- CDN利用
- リポジトリ同梱

ただし、CDN依存は可用性とバージョン固定の問題があるため、
公開安定性を優先するなら本番は同梱寄りが妥当。
この点は実装時の方針決定事項とする。

## 8. ユーザーフロー
1. ユーザーがアプリにアクセス
2. マイク利用許可を求める
3. ユーザーが録音開始
4. ユーザーが手動で録音停止
5. アプリが音声データを前処理
6. 分類モデルで推論
7. 「おじさん寄り / 上品寄り」を表示
8. 履歴保存設定に応じて結果を保存
9. 必要に応じて録音音声も保存
10. ユーザーは履歴一覧を参照・削除できる

## 9. 機能要件

### 9.1 録音機能
**要件**
- 入力方法は **マイク録音のみ**
- 録音開始後、ユーザーが **手動停止** する
- 録音中は UI 上で録音状態を明示する
- マイク権限が拒否された場合はエラー表示する

**補足**
手動停止方式は実装は単純だが、
無音が長く混ざると分類精度に悪影響が出やすい。
そのため、将来的には以下の導入余地がある。
- 無音区間の自動トリミング
- 最大録音時間制限
- 録音時間下限チェック

初版では必須ではないが、推奨事項とする。

### 9.2 判定機能
**要件**
- 録音完了後に推論を実行する
- 出力は 2値のみ
  - おじさん寄り
  - 上品寄り
- 初版では信頼度や連続スコアは表示しない

**注意**
2値のみ表示は UX 上は簡潔だが、
判定根拠が見えず、誤判定時の納得感が低い。
そのため内部的にはスコアを持っていてもよいが、UI表示はしない構成が現実的。

### 9.3 履歴機能
**要件**
- 判定履歴一覧を表示する
- 各履歴には最低限以下を保持する
  - 実行日時
  - 判定結果
  - 録音保存有無
- 履歴の個別削除
- 履歴の全件削除

**履歴表示例**
- 2026-04-08 14:23 / おじさん寄り
- 2026-04-08 14:25 / 上品寄り

### 9.4 録音保存機能
**要件**
- 録音音声を保存するかどうかをユーザーが選択できる
- 保存しない場合は、判定処理後に音声データを破棄する
- 保存する場合は `LocalStorage` に保存する

**注意**
音声バイナリを `LocalStorage` に保存するのは容量制約に弱い。
短い音声でも件数が増えるとすぐ限界に達する可能性が高い。

そのため実装では以下を推奨する。
- デフォルトは「音声を保存しない」
- 保存時は短時間音声のみ対象
- 保存件数上限を設ける
- 容量超過時は明示的にエラー表示する

初版の保存機能は「便利機能」ではなく、**制限つき機能** として扱うべき。

### 9.5 モデル未設定時の挙動
**要件**
初版公開時に学習済みモデルが未用意の場合は、以下の挙動とする。
- アプリ自体は公開する
- 判定機能は **準備中表示** にする
- 録音ボタンまたは判定ボタンは無効化する
- 「モデル未設定」または「準備中」と明示する

**禁止事項**
- ダミー判定を返さない
- 疑似ランダムな結果を返さない

この判断は妥当です。ダミー判定は面白さはあっても、仕様上はノイズです。

## 11. データ仕様

### 11.1 保存先
- `LocalStorage`

### 11.2 保存対象
**必須**
- 履歴データ
- ユーザー設定
  - 音声保存ON/OFF
  - 必要に応じてその他UI設定

**任意**
- 録音音声データ

### 11.3 保存データ例
```json
{
  "settings": {
    "saveAudio": false
  },
  "history": [
    {
      "id": "uuid",
      "timestamp": "2026-04-08T14:23:11+09:00",
      "result": "ojisan",
      "audioSaved": false
    }
  ]
}
```

音声を保存する場合の例:

```json
{
  "id": "uuid",
  "timestamp": "2026-04-08T14:23:11+09:00",
  "result": "johin",
  "audioSaved": true,
  "audioData": "base64..."
}
```

### 11.4 データ保持方針
- 端末ローカル保存のみ
- 他端末と同期しない
- ブラウザデータ削除時に消える可能性がある

## 12. 非機能要件

### 12.1 公開要件
- GitHub Pages で配信可能であること
- 静的ファイルのみで構成可能であること

### 12.2 性能要件
- 一般的なスマホ・PCブラウザで動作すること
- 録音停止から結果表示まで過度に待たせないこと
- UI がフリーズしないこと

### 12.3 可用性
- 外部 API 依存を持たないことを基本とする
- CDNを利用する場合は代替手段を検討すること

### 12.4 保守性
- モデル差し替えが可能な構成にする
- 履歴データスキーマにバージョン管理余地を持たせる

### 12.5 セキュリティ・プライバシー
- 録音前にマイク権限取得
- 保存は端末内のみ
- 音声保存が ON の場合、UI 上で明示する
- 収集データを外部送信しない構成を基本とする

## 13. エラー処理
以下の場合はユーザーに明示的なメッセージを表示する。
- マイク権限拒否
- 録音失敗
- モデル未読込
- モデル推論失敗
- 保存失敗
- LocalStorage 容量超過
- 非対応ブラウザ

## 14. 受け入れ条件
以下を満たした場合、初版の最低要件を満たす。
- GitHub Pages でアクセスできる
- マイク録音が開始・停止できる
- モデルがある場合、録音後に2値判定が表示される
- モデルがない場合、準備中表示になる
- 履歴が LocalStorage に保存される
- 履歴一覧を表示・削除できる
- 音声保存ON/OFFが切り替えられる
- 音声保存OFF時、音声本体を保存しない

## 16. 実装上の懸念点
この仕様には、いくつか現実的なリスクがある。

### 16.1 ラベル品質の問題
「おじさん寄り / 上品寄り」は主観的で、多数決でもラベル境界が曖昧になりやすい。
そのため、精度評価は想像以上に難しい。

### 16.2 `LocalStorage` に音声を入れる設計の弱さ
保存先を `LocalStorage` に限定したため、音声保存はかなり窮屈。
履歴だけ保存して音声は原則保存しない運用の方が堅実。

### 16.3 2値表示だけだと説明性が低い
UI を簡潔にした分、誤判定時に「なぜそうなったか」が見えない。
将来的には内部スコア保持だけでもしておく方が拡張しやすい。

### 16.4 手動停止録音はノイズ混入に弱い
録音の頭や末尾に無音・雑音が入りやすく、分類精度を落とす可能性がある。

## 17. 現時点の推奨実装方針
未定事項は残っているが、初版としては次が最も堅い。
- フロントエンドのみ
- マイク録音のみ
- 手動停止
- ブラウザ内推論
- 2値表示
- 履歴保存あり
- 音声保存は任意、ただし既定 OFF
- モデル未設定時は準備中表示
- 本番依存はなるべく同梱
